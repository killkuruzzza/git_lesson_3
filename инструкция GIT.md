# **Инструкция для работы с Git и удалёнными репозиториями**

## Что такое Git?

**Git - это одна из реализаций распределённых систем контроля версий, имеющая как и локальные, так и удалённые репозитории.** Является самой популярной реализацией систем контроля версий в мире.

## **Начало работы. Подготовка репозитория.**

1. На рабочем столе создаем папку.
2. Запускаем VS Code.
3. Открываем в левом верхнем углу проводник.
4. Открываем меню вид. Выбираем терминал.
5. Для того чтобы проверить что git установлен и какая версия используется запускаем команду **__git version__**.
6. Для инициализации git используем команду **__git init__**
7. Проверяем статус нашей папки **__git status__**.
8. Создаем новый файл.
9. Если еще раз запустить **__git status__** увидим что появились неотслеживаемые файлы.

# **Команды**

## *git add* 
Это одна из ключевых Git-команд, перемещает из рабочей папки в область индексирования предназначенное для следующей фиксации содержимое.

## *git commit*
Команда git commit берет все содержимое файлов, проиндексированное с помощью команды git add, и записывает в базу данных новый постоянный снимок состояния, а затем сдвигает на этот снимок указатель ветки.

## *git log*
Команда git log используется для отображения достижимой записанной истории проекта, начиная с самого свежего оммита. По умолчанию она показывает только историю ветки, в которой вы сейчас находитесь, но можно настроить ее на вывод данных сразу о нескольких ветках. Кроме того, ее можно применять для индикации различий между ветками на уровне коммитов.

## *ggit checkout*
Команда git checkout используется для перехода в другую ветку и выгрузки содержимого веток в рабочую папку.

## *git diff*
Команда git diff служит для вычисления разницы между любыми двумя деревьями. Это может быть разница между рабочей папкой и областью индексирования (собственно команда git diff), между областью индексирования и последним
коммитом (git diff --staged) или между двумя коммитами (git diff master qbranchB).

## *git branch*
Команда git branch, по сути, представляет собой инструмент управления ветками. Она умеет выводить на экран список имеющихся у вас веток, оздавать новые, а также удалять и переименовывать их.

## *git status*
Команда git status показывает состояние файлов в рабочей папке и области индексирования. С ее помощью вы можете узнать, какие файлы подверглись изменениям, но пока не индексировались, а какие уже индексировались, но пока не зафиксированы. При обычном формате вывода также выводятся подсказки о том, как изменить состояние файлов.


## **Продолжение работы.**

* запускаем git add имя_файла

* внесем изменения в файл
* сохраним ctrl+s или автосохранение
* опять git status
* запускаем git add имя_файла
* git commit -m " новый нкомментарий"
* git log
* git checkout имя commita

## **Создание новой ветки**

* Для создания новой ветки используем команду *git branch* и добавляем имя ветки.
* Проверяем что ветка создана (набирая команду *git branch*) 
* Текущая ветка, на которой мы находимся, обозначена * и выделена зеленым цветом.
* Для того, чтобы перейти на вновь созданную ветку набираем команду *git checkout* и имя новой ветки.  
Проверяем что мы перешли на нужную ветку (набирая команду *git branch*)  
* Работаем в новой ветке.

## **Слияние веток**  

* Перед слиянием веток нужно перейти на основную ветку, в которую будет добавляться информация из текущей ветки (git checkout master)  
* Проверяем что мы перешли на нужную ветку (набирая команду *git branch*)
* Для слияния веток используем команду *git merge* и название ветки, с которой будет добавляться информация в основную ветку.  

## **Удаление веток**  

Для удаление ветки используем команду *git branch -d* и имя ветки, которую надо удалить.  
Не забываем проверять с помощью команды *git branch* результаты проделанных действий. 

## **Слияние веток с конфликтом**
Если информация в новой ветке не только была добавлена новая но и была произведена правка уже существующей информации, то при слиянии веток возникнет конфликт, который разрешается вручную.  
Если в новой ветке не только была добавлена новая информация, но и была изменена уже существующей информации, то при слиянии веток возникнет конфликт, который разрешается вручную.  

## **Работа с удаленными репозиториями**  

Чтобы иметь возможность совместной работы над каким-либо Git-проектом, необходимо знать, как управлять удалёнными репозиториями. Удалённые репозитории — это модификации проекта, которые хранятся в интернете или ещё где-то в сети.

git clone — создание копии (удаленного) репозитория

git pull — забираем изменения из центрального репозитория

git push — вносим изменения в удаленный репозиторий

Делаем fork интересующего нас репозитория
Делаем git clone для нашей версии этого репозитория
Создаем ветку с предлагаемыми изменениями
Производим все изменения только в этой ветке
Отправляем эти изменения на свой аккаунт команда push
В окне на ГитХаб появляется возможность отправить pull request
